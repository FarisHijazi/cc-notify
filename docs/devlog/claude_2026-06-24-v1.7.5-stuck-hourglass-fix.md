# v1.7.5 вЂ” fix tabs frozen on вЏі after a session is done

User saw several finished tmux-in-Cursor sessions stuck showing the вЏі hourglass tab,
and focusing them didn't fix it.

## Investigation (non-destructive)

- Read `/tmp/cc-notify/*.tab`: the files **literally contained `вЏі`** for the done
  sessions в†’ not extension staleness; the `Stop` write never landed.
- Pattern: long-running tmux sessions stuck; tiny/ended ones fine. All stuck ones

- `tmux list-clients`: the route ttys are still attached (so it's not a detached
  client). `cc_detect_terminal` returns `vscode` *now* вЂ” but it's flaky under tmux.
- Inspected the **deployed** code (`~/.claude/plugins/cache/farishijazi-plugins/
  cc-notify/1.7.1/hooks/cc-notify.sh`): the tab write is
  `if term=vscode && shell_pids; then cc_write_tab; fi` with **no else**
  (`cc_set_status` count = 0). Confirmed root cause.

## Root cause

Two writers, mismatched gating (see @LESSONS.md #18):
- mid-turn **вЏі** comes from the **ungated** cheap path (`cc_set_status` in
  `cc-capture-window.sh`) вЂ” always succeeds.
- the **done/outcome** write on `Stop` goes through `cc-notify.sh`'s `cc_write_tab`,
  **gated on `term=vscode`** which flakes under tmux. When it flakes, the full write
  is skipped entirely and the last-written вЏі freezes.

## Fix

`hooks/cc-notify.sh`: added an `else` fallback to the tab-write block вЂ”
`else ( cc_set_status "$sid" "$status_emoji" & )`. The cheap path has no term/pid
dependency, so the outcome emoji is always swapped onto the existing `.tab` even when
detection fails. вЏі can never outlive the turn now.
