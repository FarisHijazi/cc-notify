# v1.7.6 ‚Äî auto-sweep so background tabs stop going stale

## Problem

A backgrounded session's terminal tab only re-renders when its terminal becomes
**active** (VS Code/Cursor API limit ‚Äî `renameWithArg`/`reapplyActiveTab` act on the
active terminal only). So when session B finishes (`Stop`: ‚è≥‚Üí‚úÖ) while you're looking
at window A, B's tab stays frozen on ‚è≥ until you focus it. The previous session built
the **sweep** (cycle every terminal via `focusNext` so each repaints) but nothing
**auto-fired** it ‚Äî the user ran it by hand, so tabs drifted stale between runs.

## What shipped

### `bin/cc-input-idle` (+ `.swift` source)
Prints `<keyboard_idle_sec> <mouse_idle_sec>`, read **live** from the OS HID state via
`CGEventSourceSecondsSinceLastEventType(.combinedSessionState, ‚Ä¶)`. No event tap, no
background daemon, **no accessibility/input-monitoring permission**, no state file ‚Äî
the kernel already tracks last-event-time per type; we just read it. Distinguishes
keyboard from mouse (combined `HIDIdleTime`/`ioreg` can't). Compiled with `swiftc -O`
on first use to `~/.cache/cc-notify/` (~36ms/run; the repo/plugin-cache dir may be RO).

> Deliberately did NOT build the `~/.last_mouse_moved.txt` file + daemon the user
> first described ‚Äî the live query is strictly simpler and stateless (matches the
> "narrow-waist / single source of truth" preference). The single source of truth is
> the HID session state itself.

### `bin/cc-sweep` ‚Äî the gated, queued trigger
Touches `/tmp/cc-notify/.sweep` (every editor window's extension watches it ‚Üí
`sweepThisWindow()`). Guards (both off with `--force`):
1. **Throttle** ‚â§1 sweep / `CC_SWEEP_THROTTLE` (10) s. State = `.sweep`'s own mtime
   (extension watches but never deletes it) ‚Üí no extra file.
2. **Keyboard guard** ‚Äî skip if a key was pressed within `CC_SWEEP_KBD_GUARD` (3) s.
   Mouse movement does NOT block (moving the mouse ‚áí not typing ‚áí safe to sweep).

A blocked sweep is **queued, not cancelled**: a single background drainer (atomic
`mkdir` lock = single-flight) polls every `CC_SWEEP_POLL` (1) s until both guards
clear, then fires once, up to `CC_SWEEP_QUEUE_MAX` (600) s. Many blocked triggers
coalesce into ONE pending sweep.

### Auto-trigger ‚Äî `cc_trigger_sweep` (cc-lib.sh)
Backgrounds `../bin/cc-sweep` (path via the function's own `BASH_SOURCE`, since cc-lib
lives in `hooks/`). Wired on **settled** events only:
- `cc-notify.sh`: right after the tab write ‚Üí covers `Stop` + every `Notification`
  type (incl. the idle tab-only case), even when the banner is suppressed.
- `cc-capture-window.sh`: `SessionStart` + `SessionEnd` only.

NOT on `PreToolUse`/`PostToolUse`/`SubagentStop`/`PreCompact` ‚Äî mid-turn churn would
flicker the active window every ~10s. Turn boundaries are enough: when B hits `Stop`,
that hook fires a global sweep ‚Üí B's background tab repaints to ‚úÖ.

## Tests (all passed)

- `cc-input-idle`: keyboard idle grew +1s/iter under `sleep 1`; mouse idle tracked
  separately ‚Äî confirms per-type, no perm prompt.
- `cc-sweep`: skip-when-typing (guard 999), fire-when-idle (guard 0), `--force` bypass.
- throttle: rapid retries ‚Üí mtime unchanged; `--force` bypasses; fires again after window.
- queue: blocked ‚Üí lock created + `.sweep` untouched; 3 more triggers ‚Üí still 1 drainer;
  after window ‚Üí drainer fired once + lock cleaned up.
- `cc_trigger_sweep`: resolves `../bin/cc-sweep` via `BASH_SOURCE` and fires (the
  earlier "FAIL" was a test artifact ‚Äî inline-defined functions have empty BASH_SOURCE).
- `bash -n` clean on all three hooks.
